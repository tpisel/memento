# ADR-0008 — Memento brief: agent-facing manifest projection

## Decision

Add a `memento brief` verb that emits the agent-facing markdown projection of the manifest. The brief, not the raw JSON, is what bootloaders inject into agent context.

- `compile` writes two artifacts: `.memento/manifest.json` (canonical, versioned) and `<vault>/_memento/brief.md` (derived, gitignored).
- `brief` reads `_memento/brief.md` if present; otherwise renders from `manifest.json`, writes the file, and emits to stdout. If `manifest.json` itself is missing, the user is directed to run `compile`.
- No `--live` flag — the disk path is the only path.

The brief lives under `_memento/` rather than at vault root because that folder is the agreed namespace for tool-relevant human-readable artifacts (ADR-0009). Obsidian's default dot-folder hiding still does not bury it. The file is gitignored to avoid silent divergence with `manifest.json`, which is the single source of truth.

### Editing guards

The brief is a derived artifact that the user should not edit by hand. Guards are layered, not locked:

1. **Frontmatter `mode: read-only`** — reuses memento's existing write-mode enforcement (ADR-0004). Any future `memento write` against the brief refuses.
2. **Top-of-file banner** — an Obsidian caution callout warning that the file is generated by `memento compile` and edits will be overwritten. The warning remains visible in Obsidian and readable as quoted Markdown elsewhere.
3. **Automatic regeneration** — the file is gitignored and regenerated on every `compile`. Edits silently disappear on the next refresh, which is the right behavior for a derived artifact.

OS-level locks (chmod, immutable bits) are deliberately not used: cross-platform fragility, and compile would need to chmod-write-chmod every refresh. The social contract plus auto-regeneration is sufficient for a single-user durable tool.

## Context

The current manifest is consumed as JSON. That JSON carries structural overhead (field names, empty `links` arrays, HTML-escape leakage like `<` and `&` from Go's default encoder) that costs agent context tokens without aiding retrieval. Markdown is the native medium for both the agent surface and the human-via-Obsidian surface; JSON remains the right surface for machine consumers (the compile pipeline, a future MCP server, future tooling).

Manifest summaries today are extracted from the first paragraph after the title heading (`internal/markdown/metadata.go:firstParagraphText`). Frontmatter `summary:` overrides that fallback. The summary discipline ("lead with the decision, not the question") is a content concern that the brief surfaces but does not enforce — see `memento-memory/what makes a good summary.md`.

Two alternatives were considered and rejected:

- **Format flag on `compile`** (e.g., `compile --format=markdown`). Rejected because `compile` mutates disk (rebuilds the index) while brief is read-only. Bundling them makes `--print --format=markdown` ambiguous about whether the on-disk artifact changes, and conflates two conceptual operations.
- **Inline rendering in the bootloader hook** (shell snippet in the AGENTS.md / CLAUDE.md injection). Rejected because the rendering is design, not plumbing — it needs tests and version-locking with the manifest code. Any future consumer (MCP server, alternate IDE integration) would otherwise re-implement the same transformation.

## Consequences

- The bootloader injects `memento brief` output in place of the raw `manifest.json` reference. Section anchors and key paths remain stable per ADR-0007 so beads tasks and commit messages can still hotlink them.
- `compile` is now responsible for both artifacts; they go stale together and refresh together. The pre-commit hook (ADR-0005) already drives manifest regeneration; the brief tags along.
- `_memento/brief.md` is added to the vault `.gitignore` injection (vault-scoped) and to `.mementoignore` so the compile pass does not index its own output. The wider `_memento/` folder is *not* fully ignored — see ADR-0009 — only specific generated files within it.
- A per-entry size signal (bytes and lines) joins the manifest schema so the brief can show approximate cost-to-load alongside each entry. Token-aware sizing is deferred until the brief itself approaches context budgets — at that point a tokenizer is justified; until then, bytes/lines are a free, model-independent proxy.
- The brief renders heading text only; slugs are derivable from text, so the agent-facing duplication is removed. A future `memento read` enhancement may accept a heading by text as a second argument and resolve to the slug live.
- `summary_stale` exposure and tag-discipline filtering are deferred to the agent-driven summarisation / write flow. Both remain in the JSON schema; the brief either drops them or annotates inline once that feature lands.
- Doc-type-specific rendering (ADRs vs. spec vs. notes) is deferred. v0 renders uniformly.

## Open questions

- A convention for *open* (unresolved) design questions alongside ADRs (which capture resolved ones) is not yet chosen. Common patterns include `design-notes/`, `proposals/`, or RFD-style numbered docs. Out of scope for this ADR; flagged for a later decision.
- The brief may carry a footer listing tool-consumed files present in `_memento/` (e.g., `writing.md`, `review.md`) once those filenames are pinned by ADR-0010. Scoped into the renderer bead.
